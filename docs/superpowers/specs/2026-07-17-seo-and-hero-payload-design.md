Branch: `seo-improvements` (4 commits ahead of `main`, unpushed)

## The problem, restated

The audit's roadmap reads as six loosely related chores. Investigation collapsed most of it into **one root cause with three symptoms**, plus a small set of genuinely independent items.

### Root cause: media queries resolve in effects, not at init

`usePrefersReducedMotion` and `CookieScrubHero`'s `isSmall` both start `false` and only flip inside `useEffect`. React runs child effects before parent effects, so `Scrub`'s one-shot fetch effect (`}, [])`, guarded by `startedRef`) always fires while both flags are still `false`. Consequences, all measured:

| Symptom | Measured | Should be |
|---|---|---|
| Reduced-motion users download frames they never see | 96 requests, **11.5 MB** | 0 |
| Mobile decode ignores `resizeWidth: 760` | **0** calls with opts, 96 without, decoded **1600×873**, **~512 MB** bitmap RAM | ~121 MB at 760px |
| Responsive frames would fetch the wrong set | n/a (would silently break) | phone gets 760px set |

The third is why this must be fixed *before* the payload work, not after: a naive responsive-frame change resolves `mobile === false` on first render, fetches the desktop set, and `startedRef` then blocks any correction. The fix would appear to do nothing.

The ~512 MB figure is the most serious number in the audit. It is a crash risk on real phones, and it is invisible to every SEO tool.

### Independent items

Real 404s, image `alt`, and deployment share nothing with the above and are planned separately.

## Decisions taken

| Decision | Choice | Why |
|---|---|---|
| Hero payload | **Responsive WebP, keep 96 frames** | Phones already downscale to 760px, so a 760px WebP set is pixel-identical to what phones see today: 11.5 MB → 2.5 MB for zero visual change. Desktop 1600px WebP: 11.5 MB → 7.6 MB. Preserves the deliberate jank-free ImageBitmap scrub. |
| Video encoding | **Rejected** | 1.2 MB (−90%) but `currentTime` scrubbing is unreliable on iOS Safari — the exact audience. Buys 1.3 MB over WebP-760 by undoing the design decision the hero exists for. |
| Fewer frames | **Rejected** | Same savings as video; costs scrub smoothness for 1.3 MB. |
| `og:image` + favicon | **Deferred** — user is making the art | `twitter:card` stays `summary`. |
| CWV measurement | **Deferred** — user is getting a PageSpeed key | Local lab numbers remain directional only. |
| Deploy | **Merge to `main` + push** | Cloudflare builds `main`. |
| AI crawler posture | **Unchanged** | Reviewed in audit 1 and deliberately kept. |

Measured encoding options, from the real frames:

| option | payload | vs today |
|---|---|---|
| JPEG 1600 (today) | 12 MB | — |
| WebP 1600 | 7.6 MB | −37% |
| **WebP 760** | **2.5 MB** | **−79%** |
| WebP 760, 48 frames | 1.2 MB | −90% |
| h264 / vp9 760 | 1.2 MB | −90% |

## Design

### 1. Resolve media queries synchronously

A single hook, `useMediaQuery(query)`, reading `window.matchMedia(query).matches` in the `useState` initialiser and subscribing for changes. `usePrefersReducedMotion` becomes a one-line wrapper over it, keeping its current call sites unchanged. `isSmall` uses it directly.

SSR safety is not a concern (`createRoot`, no hydration), but the initialiser guards `typeof window` regardless so the prerender's Node-side never touches it.

The result: first render already knows both answers. `<Scrub>` never mounts for reduced-motion users, and when it does mount it knows whether it's a phone.

### 2. Cancel in-flight frame fetches

`Scrub`'s decode loop takes an `AbortController`; the effect's cleanup aborts it. Fixes the StrictMode double-mount case and any future unmount, and means an aborted load can't land bitmaps into a dead component.

`startedRef` stays: it exists to survive StrictMode's double-mount, which is still real.

### 3. Serve responsive WebP frames

`public/cookie/f_*.jpg` (12 MB) is replaced by two generated sets:

```
public/cookie/760/f_001.webp … f_096.webp    2.5 MB
public/cookie/1600/f_001.webp … f_096.webp   7.6 MB
```

`framePath(i)` takes the resolved `mobile` flag and picks the set. Each client fetches exactly one set. Repo grows by nothing meaningful (12 MB out, 10.1 MB in); served bytes drop 79% on mobile, 34% on desktop.

Generated by a committed script (`scripts/encode-frames.mjs`) rather than by hand, so the sets are reproducible from source art and the quality setting is reviewable. Encoding runs once, output is committed — it does **not** join the build, which already carries puppeteer.

WebP support is universal among browsers that have `createImageBitmap` at all (Safari 15+), which the hero already requires. No JPEG fallback; the old set is deleted.

`decodeOpts` keeps `resizeWidth: 760` on mobile — now that it will actually run, it also guards against a phone with a large viewport picking the 1600 set.

### 4. Real 404s

Add `public/404.html`. Cloudflare Pages serves it with a real 404 status for unmatched paths and stops falling back to `index.html` with 200. Safe here: the site is one route and navigation is same-document hash anchors, so nothing depends on the catch-all.

Content: minimal, styled to match, linking home. Not a React page — it must work as a static document.

### 5. Image alt attributes

- Decorative (crumbs, background figures, the frame canvas) → `alt=""`, which is the correct answer and not the same as omitting it.
- Meaningful (mascot, widget screenshots, `figures/`) → short factual alt.
- The hero canvas is already `aria-hidden`; the reveal wordmark carries the meaning. Verify rather than add.

### 6. Deploy

Merge `seo-improvements` to `main`, push, watch the Pages build (first run pulls Chromium), then verify on production: raw HTML has body text, `sitemap.xml` is `application/xml`, `/nonexistent` returns 404, and **Turnstile mounts and the demo form submits** — the one surface never verified locally.

## Verification

Each phase is verified by measurement, not inspection:

| Phase | Check |
|---|---|
| 1–2 | Instrumented puppeteer run: reduced-motion → **0** frame requests; phone → 96 `createImageBitmap` calls **with** `resizeWidth`, decoded 760×415 |
| 3 | Phone fetches only `/cookie/760/`, total < 3 MB; desktop only `/cookie/1600/`; hero still scrubs |
| 4 | `curl -o /dev/null -w '%{http_code}' /nonexistent` → 404; `/` → 200 |
| 5 | Every `<img>` has an `alt` attribute; no non-empty alt on decorative images |
| 6 | Production: body text > 5,000 chars, sitemap `application/xml`, Turnstile mounts |

Existing suite (7 tests) and `tsc` must stay green throughout. Visual parity re-checked with the screenshot diff already used for prerendering.

## Risks

- **Visual regression in the hero.** WebP q78 is lossy at a different point than the source JPEG. Mitigated by a screenshot diff of the scrub at several progress points; q78 is adjustable.
- **Turnstile.** Still unverifiable until deployed. It is the one thing that could make the deploy a rollback rather than a win.
- **Pages build.** Puppeteer's Chromium download could fail in the Pages container. The build fails loudly by design; the fallback is reverting the `build` script line, not a silent skip.

## Out of scope

`og:image`, favicon, PageSpeed key (all with the user); `llms.txt`; FAQPage/HowTo schema; framework migration; AI-crawler posture.
